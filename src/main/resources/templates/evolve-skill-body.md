## Plugin house conventions

- `build.gradle.kts` is the single source of truth for the version; `processResources` syncs it
  into `plugin.json` ‚Äî never hand-edit the version in `plugin.json`.
- Entry point implements `DynamicPlugin` (`register(context)` / `dispose()`); UI is Compose
  Multiplatform (never Android APIs); state lives in ViewModels exposing `StateFlow`.
- Providers from `PluginContext` (splitViewOperations, notificationProvider, ‚Ä¶) may be null ‚Äî
  handle gracefully, never crash.
- `boss-plugin-api` is `compileOnly` (the host provides it at runtime). Do not add it as
  `implementation`, and avoid new runtime dependencies unless unavoidable (thin jar).
- All Kotlin files end with a newline.
- While iterating locally keep ONE version and overwrite the same jar; bump the version only in
  the final commit.

## Evolution loop

1. Understand the requested change. Read the relevant sources in this repo first.
2. Implement the change.
3. Build: `./gradlew buildPluginJar` ‚Üí jar lands in `build/libs/` (ignore `-thin`/`-all` jars).
4. **Hot-reload into the running BOSS instance**: call the MCP tool `evolver_hot_reload`
   (exposed by the host's `boss` MCP server ‚Äî typically `mcp__boss__evolver_hot_reload`; if the
   server name differs, check `$BOSS_MCP_SERVER`) with:
   - `plugin_id`: `@@PLUGIN_ID@@`
   - `jar_path`: the absolute path of the jar you just built
   It copies the jar into `@@PLUGINS_DIR@@` (replacing older jars of this plugin) and live-loads
   it ‚Äî no app restart. Fallback if the tool is unavailable: copy the jar there yourself and ask
   the user to use the panel's ‚ãÆ ‚Üí Reload Plugin.
5. Verify: call `evolver_probe` with `plugin_id` `@@PLUGIN_ID@@` ‚Äî confirm the new version is
   loaded and healthy and no errors appear in the log tail. Ask the user to exercise the UI when
   behavior can't be verified from logs alone.
6. Iterate 2‚Äì5 until the evolution works.

## Ship it (required once the evolution is verified)

Create a PR ‚Äî do NOT push `main` directly (pushing `main` triggers a store release):

```bash
git checkout -b evolve/<short-topic>
git add -A
git commit -m "<type>: <what evolved>"   # feat/fix/refactor‚Ä¶
git push -u origin evolve/<short-topic>
gh pr create --fill --body "<what changed, why, and how it was verified (hot-reloaded live via Tool Evolver)>"
```

If `origin` is missing or `gh` is unauthenticated, stop after the commit and tell the user.
